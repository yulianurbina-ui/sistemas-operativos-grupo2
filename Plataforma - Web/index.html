Fase 1

Componente 1: Virtualización

Se realizó el despliegue de dos hipervisores aislados utilizando almacenamiento controlado. Ambas máquinas se interconectaron por medio de una interfaz de red interna estática.

Evidencia 1.1: Verificación estructural del disco mapeado con particionado manual (VM Consola).
Evidencia 1.2: Particionamiento local de la máquina gráfica (Ubuntu MATE).
Evidencia 1.3: Verificación de Particiones Activas en Producción.
Evidencia 1.4: Configuración de Firewall para Servicio SSH.
Evidencia 1.5: Configuración y Verificación de Red Interna.
Evidencia 1.6: Túnel seguro SSH activo y autenticado en puerto 22.

Comandos obligatorios de Verificación ejecutados

bash - VM Consola 
# Comprobación de la distribución instalada
cat /etc/os-release

# Análisis del espacio y esquema de particiones manuales
lsblk

# Configuración de interfaces y asignación de IPs
ip a

# Conectividad de red interna cruzada (Ping hacia el nodo adyacente)
ping -c 4 [IP_VM_OBJETIVO]
Fase 2

Componente 2: Aislamiento por Contenedores Docker

Instalación nativa de Docker Engine en Ubuntu MATE. Se tomaron los archivos base proveídos (`Dockerfile_fedora_rocky_alma_ACTUALIZADO`) y se segmentaron en dos entornos independientes fundamentados en la imagen rockylinux:9.

Orquestación de Capas: Frontend y Backend

Utilizando el script del backend en Python (server.py en puerto 5000) y el servidor Nginx, se vinculó el archivo docker-compose.yml configurando volúmenes de persistencia.

bash - Docker Engine 
# Desplegar los servicios en segundo plano (Modo Detached)
docker compose up -d

# Validar el ciclo de vida y puertos expuestos de los contenedores
docker ps

# Diagnóstico de respuestas HTTP locales
curl http://localhost
curl http://localhost:5000

Instalacion de Docker Engine - Ubuntu Mate - ( VMWARE )

Evidencia 2.1: Construcción y levantamiento de servicios vía Docker Compose.
Evidencia 2.2: Despliegue exitoso del microservicio Backend (Python en puerto 5000)
Evidencia 2.3: Accesibilidad y ruteo del microservicio Frontend (Nginx en puerto 8080)
Fase 3

Componente 3: Kubernetes

Se implementó un cluster local de Kubernetes usandoMinikube obre una máquina virtual Ubuntu. Se desplegó un servidor web Nginx mediante manifiestos YAML, se expuso el servicio con NodePort y se demostró el escalado manual de réplicas.

Evidencia 3.1: Verificación de entorno e instalación de Minikube y Kubectl.
Evidencia 3.2: Despliegue de manifiestos YAML y verificación de las 2 réplicas iniciales.
Evidencia 3.3: Acceso web exitoso al servicio Nginx expuesto por NodePort.
Evidencia 3.4: Escalado horizontal dinámico de 2 a 3 réplicas sin tiempo de inactividad.

William F. Ibargüen
C1: Virtualización

"El laboratorio demostró con éxito la ventaja técnica de separar las particiones de datos (/home) de las del sistema (/) para garantizar la seguridad de la información. Asimismo, comprobó la alta eficiencia de los sistemas de servidor basados puramente en consola al ahorrar recursos de hardware (RAM y CPU), y la viabilidad de administrarlos de forma 100% remota y segura a través de una red"

Nicolas Barona
C2: Docker Engine

"La transición de máquinas virtuales a contenedores fue un punto de inflexión en el proyecto. Trabajar la orquestación con Docker Compose me permitió entender en la práctica cómo el desacoplamiento de microservicios (Nginx y Python) optimiza drásticamente los recursos del host y agiliza los despliegues"

Estevan F. Gómez
C3: K8s Minikube

"Orquestar servicios con Kubernetes cambió mi perspectiva sobre la alta disponibilidad. Implementar Minikube y ver en tiempo real cómo el clúster es capaz de escalar réplicas dinámicamente usando manifiestos declarativos YAML evidencia el verdadero poder del Cloud Computing"

Yulian F. Urbina
